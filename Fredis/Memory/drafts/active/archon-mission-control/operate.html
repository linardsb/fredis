how the three fit together

mission control is the board — what's outstanding across your repos. archon is the engine — it runs a task in an isolated git worktree and opens a draft pull request. fredis is the chat front-end onto the same engine. this is the day-to-day operating procedure for the three together.

00

one-time setup

after v1 build
  1. 1
    install archon

    build the engine from the local clone.

    cd ~/Desktop/cloned_repos/archon/remote-coding-agent bun install && bun run build

    the server starts and GET /api/workflows returns json on port 3090.

  2. 2
    pin it to localhost

    archon's api has no auth and binds all interfaces — bind it to 127.0.0.1 or firewall port 3090 so it isn't an open code-execution surface on your network.

  3. 3
    configure claude auth

    same subscription as fredis is fine — but clear any stale CLAUDE_CODE_OAUTH_TOKEN env var first (it silently shadows a good login).

    a trivial run completes end-to-end on a throwaway repo.

  4. 4
    serve mission control over http

    the live overlay and @delegate make network calls, which file:// blocks — serve this folder over http.

    cd ~/Desktop/mission-control python3 -m http.server 5000

    the board opens at http://localhost:5000.

  5. 5
    wire @delegate

    point the board's delegate action at archon's api (phase 1 of the plan). one-time, then it's a button.

01

start of day

after v1 build
  1. 1
    start archon's server

    one command (or a login item to automate it). its sqlite keeps all run history across restarts. you do not need it running 24/7 — only while you work; a run in flight needs the machine up until it finishes.

  2. 2
    open the board

    visit http://localhost:5000. the static file:// board still opens for plain viewing, just without live data or delegate.

02

see what's outstanding

works today
  1. 1
    read the board

    status across every tracked repo — planned, in-progress, done, blocked — assigned from evidence, never assumption. the filter row narrows to one status.

  2. 2
    refresh from evidence

    run the scan (a read-only scanner per repo + a separate verifier on the done-claims). this stays your planning source of truth — unchanged by the archon work.

    /mission sync

    the board reloads; "done" reflects real git / PR evidence.

03

delegate a task

after v1 build
  1. 1
    pick an item, click @delegate

    the board fires a workflow run on that item's repo.

  2. 2
    choose a workflow

    start with archon-assist (read-only, safest), then archon-fix-github-issue, then the rest (pr review, interactive-prd, adversarial-dev).

  3. 3
    archon isolates the work

    it spins a dedicated git worktree on that repo — only your other repos (throwaway → email hub), never the fredis repo or the vault. the item flips to "delegated / running" live.

04

watch and approve

after v1 build
  1. 1
    drill into archon's web ui

    watch the run node-by-node and read live logs. the board gives the at-a-glance badge; archon's ui is the detail view. (it's archon's own look, not saulera.)

  2. 2
    approve a human-in-the-loop pause

    workflows like interactive-prd pause for your input — approve or send feedback in archon's ui to resume the run.

05

review the result

after v1 build scan reconcile works today
  1. 1
    review the draft pull request

    the run opens a draft PR — nothing is auto-pushed to main or sent anywhere. review it, then merge or send it back.

  2. 2
    reconcile the board

    the scan sees the draft PR as evidence and promotes the item. branch-name links the run back to the board item.

    /mission sync

    the delegated item moves to its evidence-based status.

06

the fredis path

after v1 build
  1. 1
    fire the same from chat / slack

    fredis drives the identical engine conversationally.

    run the fix-issue workflow on email hub issue 42

    fredis launches the run, monitors it, and reports back the draft PR — without touching the fredis repo.

non-negotiable guardrails

  • archon runs only on other repos (throwaway → email hub) — never the fredis repo or the vault.
  • draft pull requests only. no auto-push to main; no node that emails, slacks, or posts.
  • archon's api is unauthenticated — keep it bound to localhost.
  • local first — the "fire from my phone" setup comes later.
  • one or two runs at a time to start; they share your claude usage.
